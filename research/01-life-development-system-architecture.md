# Архитектура системы развития человека через практики

## 1. Архитектура системы

### Концептуальная модель

Система строится вокруг **цикла развития** — непрерывного процесса наблюдения, анализа, эксперимента и корректировки. В центре — **модель жизни пользователя** (Life Model), которая агрегирует все данные и является основой для принятия решений.

### Слои системы

**Layer 1: Data Ingestion** — сбор данных из всех источников. Каждый источник имеет адаптер, который нормализует данные в единый формат событий (Life Events).

**Layer 2: Life Model** — ядро системы. Хранит и обновляет модель жизни пользователя. Это не просто база данных, а семантическая модель, которая понимает связи между событиями, практиками и результатами.

**Layer 3: AI Agents** — команда агентов, каждый со своей специализацией. Работают с Life Model, генерируют insights и рекомендации.

**Layer 4: Marketplace** — каталог внешних практик с возможностью бронирования и оплаты.

**Layer 5: Dev Team** — AI-команда разработки, которая создаёт новые инструменты внутри системы.

**Layer 6: Interface** — точки взаимодействия с пользователем.

## 2. Роли AI-агентов

### Основная команда агентов

```
┌─────────────────────────────────────────────────────────────┐
│                    ORCHESTRATOR                               │
│     Координирует работу всех агентов, управляет приоритетами │
└──────────┬──────────────────────────────────────┬────────────┘
           │                                      │
    ┌──────┴──────┐                        ┌──────┴──────┐
    │  INNER TEAM │                        │ OUTER TEAM  │
    │ (анализ и   │                        │ (внешний    │
    │  стратегия) │                        │  мир)       │
    └─────────────┘                        └─────────────┘
```

#### Inner Team (работа с внутренним миром пользователя)

**1. Observer (Наблюдатель)**
- Непрерывно собирает и структурирует данные
- Выявляет паттерны в поведении
- Замечает аномалии и изменения
- Формирует "дневной отчёт" о состоянии
- *Входы:* все источники данных
- *Выходы:* структурированные наблюдения, паттерны, аномалии

**2. Analyst (Аналитик)**
- Анализирует связи между практиками и результатами
- Строит корреляции: "когда ты делаешь X, через N дней Y улучшается"
- Проверяет гипотезы статистически
- Выявляет причинно-следственные связи (насколько возможно)
- *Входы:* наблюдения Observer, исторические данные
- *Выходы:* аналитические отчёты, подтверждённые/опровергнутые гипотезы

**3. Strategist (Стратег развития)**
- Формирует долгосрочную стратегию развития
- Определяет приоритеты: какие области жизни требуют внимания
- Балансирует между сферами (здоровье, работа, обучение, отношения)
- Адаптирует стратегию на основе обратной связи
- *Входы:* цели пользователя, аналитика, текущее состояние
- *Выходы:* стратегический план, приоритеты, рекомендации по фокусу

**4. Coach (Коуч / Психолог)**
- Ведёт диалог с пользователем
- Помогает с рефлексией
- Замечает когнитивные искажения
- Поддерживает мотивацию
- Помогает осознать сопротивление изменениям
- *Входы:* диалоги, дневники, эмоциональные данные
- *Выходы:* инсайты, вопросы для рефлексии, поддержка

**5. Practice Designer (Дизайнер практик)**
- Проектирует персональные практики
- Адаптирует существующие практики под контекст пользователя
- Определяет оптимальную интенсивность, длительность, частоту
- Создаёт прогрессии (от простого к сложному)
- *Входы:* стратегия, текущие возможности, предпочтения
- *Выходы:* описания практик, расписания, критерии успеха

#### Outer Team (работа с внешним миром)

**6. Scout (Скаут / Исследователь)**
- Ищет внешние практики, курсы, программы
- Оценивает их качество и релевантность
- Мониторит новые возможности
- Собирает отзывы и рейтинги
- *Входы:* запросы от Strategist, каталог маркетплейса
- *Выходы:* подборки практик, оценки, рекомендации

**7. Planner (Логистик / Планировщик)**
- Планирует расписание практик
- Бронирует внешние активности
- Организует логистику (перелёты, проживание)
- Управляет конфликтами в расписании
- *Входы:* выбранные практики, календарь, бюджет
- *Выходы:* план-расписание, бронирования, напоминания

#### Meta Team (развитие самой системы)

**8. System Architect (Архитектор системы)**
- Определяет, когда нужен новый инструмент
- Формирует требования к инструменту
- Приоритизирует разработку
- *Входы:* узкие места системы, запросы пользователя, идеи агентов
- *Выходы:* спецификации инструментов, приоритеты разработки

### Модель взаимодействия агентов

```
Observer ──наблюдения──▶ Analyst ──инсайты──▶ Strategist
                                                  │
                              ┌────────────────────┤
                              ▼                    ▼
                           Coach            Practice Designer
                              │                    │
                              ▼                    ▼
                        Пользователь          Scout ──▶ Planner
                        (рефлексия)           (поиск)   (booking)

Все агенты ──обратная связь──▶ System Architect ──▶ Dev Team
```

Агенты работают **асинхронно** и **событийно**:
- Observer работает постоянно (при поступлении данных)
- Analyst запускается периодически или по триггеру Observer
- Strategist пересматривает стратегию еженедельно или по значимым событиям
- Coach доступен по запросу и инициирует контакт при необходимости
- Scout ищет практики по запросу Strategist
- Planner активируется при выборе практики

### Нормализация в Life Events

Все данные приводятся к формату **Life Event**:

```
LifeEvent {
  id: uuid
  timestamp: datetime
  source: string          // "garmin", "journal", "manual"
  category: string        // "health", "work", "learning", "emotion"
  type: string            // "sleep_session", "workout", "mood_entry"
  data: JSON              // специфичные данные
  metrics: {key: value}   // извлечённые метрики
  context: string         // AI-обогащённый контекст
}
```

## 5. Цикл изменения практик

### Основной цикл (Development Loop)

```

    │                                                      │
    ▼                                                      │
┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐      │

                                              │            │
                                              ▼            │

### Детализация каждого этапа

**1. OBSERVE (Наблюдение)** — Observer Agent
- Сбор данных из всех источников
- Фиксация текущего состояния
- Выявление отклонений от нормы
- *Длительность:* непрерывно
- *Выход:* "Заметил: уровень энергии падает после 15:00 последние 5 дней"

**2. ANALYZE (Анализ)** — Analyst Agent
- Поиск причин наблюдаемых паттернов
- Корреляция с другими данными
- Сравнение с историческими данными
- *Длительность:* минуты (автоматический) или по запросу
- *Выход:* "Падение энергии коррелирует с поздним обедом (после 14:00) и отсутствием прогулки"

**3. HYPOTHESIZE (Гипотеза)** — Strategist Agent + Coach Agent
- Формулировка гипотезы об улучшении
- Оценка потенциального эффекта
- Проверка на конфликт с другими практиками
- *Длительность:* часы (с участием пользователя)
- *Выход:* "Гипотеза: обед до 13:00 + 15-минутная прогулка после → энергия во второй половине дня вырастет на 20%"

**4. DESIGN (Дизайн практики)** — Practice Designer Agent
- Проектирование конкретных действий
- Определение длительности, частоты, интенсивности
- Определение метрик успеха
- Создание напоминаний и триггеров
- *Длительность:* часы
- *Выход:*
  ```
  Практика: "Energy Lunch Protocol"
  Действия:
    - Обед не позже 13:00
    - Прогулка 15 мин после обеда
  Длительность эксперимента: 14 дней
  Метрика успеха: средняя энергия 15:00-18:00 > 6/10
  ```

**5. EXPERIMENT (Эксперимент)** — Пользователь + система
- Пользователь выполняет практику
- Система напоминает, трекает, поддерживает
- Coach помогает с мотивацией и сопротивлением
- *Длительность:* дни-недели
- *Мониторинг:* Observer фиксирует compliance и результаты

**6. MEASURE (Измерение)** — Analyst Agent
- Сбор метрик за период эксперимента
- Сравнение с baseline (до эксперимента)
- Статистическая значимость изменений
- *Длительность:* автоматически по окончании периода
- *Выход:* "Средняя энергия 15:00-18:00: было 4.2, стало 6.8. Compliance: 11 из 14 дней"

**7. EVALUATE (Оценка)** — Strategist Agent
- Принятие решения: принять / модифицировать / отклонить практику
- Обновление Life Model
- Запуск следующего цикла
- *Решения:*
  - **Принять** → практика становится постоянной, мониторинг продолжается
  - **Модифицировать** → новый цикл с изменённой практикой
  - **Отклонить** → фиксация "не сработало", альтернативная гипотеза

### Временные масштабы циклов

```
Микроцикл (дни):       привычка → трекинг → корректировка
Мезоцикл (недели):     эксперимент → измерение → решение
Макроцикл (месяцы):    стратегия → реализация → пересмотр
Метацикл (кварталы):   жизненное направление → большие изменения
```

### Триггеры запуска цикла

| Триггер | Пример | Кто инициирует |
|---------|--------|---------------|
| Аномалия в данных | Энергия упала на 30% за неделю | Observer |
| Цель не двигается | Прогресс по марафону стоит 3 недели | Strategist |
| Запрос пользователя | "Хочу улучшить сон" | Пользователь |
| Завершение эксперимента | 14-дневный эксперимент окончен | Analyst |
| Внешнее событие | Смена сезона, отпуск, новая работа | Observer |
| Открытие скаутом | Найдена программа, идеально подходящая под цели | Scout |

## 6. Цикл создания инструментов

```
┌─────────────┐     ┌─────────────┐     ┌──────────────┐
│  Потребность │────▶│  Валидация  │────▶│ Спецификация │
│  (сигнал)   │     │  (нужно ли?)│     │ (что именно) │
└─────────────┘     └─────────────┘     └──────┬───────┘
                                               │
                    ┌──────────────────────────┘
                    ▼
              ┌─────────────┐     ┌─────────────┐
              │  Разработка │────▶│  Тестирование│
              │  (Dev Team) │     │  + Review    │
              └─────────────┘     └──────┬───────┘
                                         │
                    ┌────────────────────┘
                    ▼
              ┌─────────────┐     ┌─────────────┐
              │  Деплой     │────▶│  Мониторинг │
              │             │     │  + Итерация  │
              └─────────────┘     └─────────────┘
```

### Детализация

**1. Потребность** — откуда приходит сигнал:
- Агент замечает: "нет инструмента для X"
- Пользователь просит: "хочу видеть Y"
- Система обнаруживает: "это повторяющееся ручное действие — можно автоматизировать"

**2. Валидация** — System Architect проверяет:
- Действительно ли это нужно? (а не одноразовый запрос)
- Можно ли решить существующими средствами?
- Какой приоритет относительно других задач?

**3. Спецификация** — Product Designer формирует:
- User story: "Как пользователь, я хочу видеть корреляцию между сном и энергией, чтобы понимать оптимальную длительность сна"
- UI mockup (текстовое описание)
- Критерии приёмки
- Необходимые данные

**4. Разработка** — Tech Lead → Developers:
- Декомпозиция на задачи
- Backend: API endpoints, сервисы, модели
- Frontend: компоненты, страницы, хуки
- Тесты

**5. Тестирование** — QA Agent:
- Unit-тесты
- Интеграционные тесты
- Проверка с реальными данными

**6. Деплой + мониторинг**:
- Автоматический деплой
- Сбор обратной связи
- Итерация при необходимости

```

│   ├── Социальное

## 9. Пример работы системы

### Сценарий: от наблюдения до изменения жизни

**Неделя 1 — OBSERVE**

Observer замечает:
> "Последние 7 дней: средняя энергия после 15:00 = 3.8/10 (обычно 5.5). Сон: среднее 6.2ч (обычно 7.1ч). Тренировок: 1 (обычно 3). Количество кофе: 4 чашки/день (обычно 2)."

**Неделя 1 — ANALYZE**

Analyst строит корреляции:
> "Обнаружена цепочка: рабочий дедлайн → поздние вечера → сон < 6.5ч → энергия падает → больше кофе → сон ещё хуже (порочный круг). Цикл начался 5 дней назад, совпадает с записью в дневнике: 'Большой релиз на работе, не успеваю'."

**Неделя 1 — HYPOTHESIZE**

Strategist формирует гипотезу:
> "Пользователь в стрессовом цикле. Два подхода: (A) Разорвать цикл через принудительное ограничение кофе после 14:00 + фиксированное время отбоя. (B) Добавить 10-минутную дыхательную практику перед сном для ускорения засыпания."

Coach инициирует диалог:
> "Заметил, что последняя неделя была интенсивной. Как ты себя чувствуешь? Замечаешь ли ты, что попал в цикл 'мало сна → кофе → ещё меньше сна'?"

Пользователь: "Да, осознаю, но дедлайн через 3 дня. Не могу позволить себе расслабиться."

Coach: "Понимаю. А что если мы не убираем интенсивность работы, но добавляем один элемент, который поможет быстрее восстанавливаться? Всего 10 минут перед сном."

**Неделя 1 — DESIGN**

Practice Designer создаёт практику:
```
Практика: "Emergency Sleep Reset"
Цель: разорвать цикл сон→кофе→сон
Действия:
  1. Кофе-стоп после 14:00 (заменить на воду/чай)
  2. 10 мин дыхательная практика 4-7-8 перед сном
  3. Телефон в авиарежим за 30 мин до сна
Длительность: 7 дней
Метрика: время засыпания, длительность сна, энергия 15:00
Сложность: минимальная (подходит для стрессового периода)
```

**Неделя 2 — EXPERIMENT + MEASURE**

Пользователь выполняет практику. Observer трекает:
```
День 1: кофе-стоп ✓, дыхание ✓, сон 6.5ч (+0.3ч)
День 2: кофе-стоп ✓, дыхание ✓, сон 6.8ч (+0.6ч)
День 3: кофе-стоп ✗, дыхание ✓, сон 6.3ч
День 4: кофе-стоп ✓, дыхание ✓, сон 7.2ч (+1.0ч) ← дедлайн пройден
День 5: кофе-стоп ✓, дыхание ✓, сон 7.5ч, энергия 15:00 = 6.2/10
День 6-7: стабильно сон 7+, энергия возвращается к норме
```

Analyst:
> "Результат: сон вырос с 6.2ч до 7.1ч за 7 дней. Энергия 15:00 восстановилась до 5.8/10. Дыхательная практика показала 100% compliance — пользователю она подходит. Кофе-стоп сложнее (86% compliance), но тоже работает."

**Неделя 2 — EVALUATE**

Strategist:
> "Практика сработала. Рекомендация: оставить дыхательную практику 4-7-8 как постоянную вечернюю практику. Кофе-стоп — включить как правило при stress level > 6/10."

**Неделя 3 — Новый цикл (более глубокий)**

Strategist, увидев успех:
> "Пользователь восприимчив к дыхательным практикам. Текущая цель — повысить стрессоустойчивость. Рекомендую исследовать более глубокие практики."

Scout находит:

## 10. Ограничения и риски

### Технические
- **Качество данных:** субъективные оценки (энергия, настроение) ненадёжны; нужна калибровка
- **Корреляция != причинность:** Analyst может находить ложные связи; нужна осторожность в выводах
- **Cold start:** в начале мало данных — система не может давать точные рекомендации
- **Интеграции:** внешние API ненадёжны, меняют формат, могут быть заблокированы

### Психологические
- **Сопротивление изменениям:** люди прокрастинируют, саботируют, забывают — система должна это учитывать, а не осуждать
- **Перегрузка рекомендациями:** слишком много предложений → паралич выбор�� → откат
- **Зависимость от системы:** пользователь может потерять способность принимать решения без AI
- **Quantification bias:** не всё измеримо, и попытка измерить всё может навредить

### Когнитивные
- **Ошибка выжившего:** система видит только данные, которые собирает; важное может быть за пределами
- **Overfitting:** система может "заточиться" под текущее состояние и не замечать необходимость радикальных изменений
- **Confirmation bias:** агенты могут подтверждать гипотезы пользователя, а не проверять их

### Этические
- **Приватность:** система хранит интимные данные о жизни; утечка критична
- **Манипуляция:** тонкая грань между "помочь" и "управлять" решениями
- **Автономия:** система должна расширять свободу выбора, а не сужать
- **Коммерческие конфликты:** маркетплейс может рекомендовать не лучшую практику, а ту, что приносит больше комиссии

### Принципы минимизации рисков
1. **Пользователь всегда принимает финальное решение** — система рекомендует, не навязывает
2. **Прозрачность:** система объясняет, почему рекомендует X, а не Y
3. **Разделение рекомендательного и коммерческого:** маркетплейс ранжирует по полезности, не по комиссии
4. **Право на "тишину":** пользователь может поставить систему на паузу
5. **Данные принадлежат пользователю:** полный экспорт, удаление, портабельность

# Спецификация: Canvas — визуальный редактор workflow команд

## Статус

Готово к разбивке на задачи. Один открытый вопрос: выбор библиотеки рендеринга.

---

## Контекст

Canvas — визуальный редактор workflow команд агентов. Заменяет TeamPage (`/teams`).
На canvas видны все команды, их агенты и workflow (именованные цепочки агентов).

Canvas определяет:
- Какие команды и агенты существуют в системе
- Какие workflow (связки агентов) доступны в каждой команде
- Кто первый получает задачу (стартовый агент workflow)
- Кому и с каким промптом передаётся задача дальше
- При каких условиях происходит переход
- Нужно ли подтверждение пользователя на каждом переходе
- Сколько раз может зациклиться задача между агентами

## 1. Страница Canvas

**Маршрут:** `/teams` — Canvas полностью заменяет TeamPage.

Страница содержит:
- Глобальный canvas со всеми командами и агентами
- Кнопка **"+ Добавить команду"** — создаёт новую команду (group node на canvas)
- Правая боковая панель — открывается при клике на узел/ребро

TeamPage (`/teams/:id`) убирается из навигации.

## 2. Технология

**Библиотека рендеринга:** `@xyflow/react` (ReactFlow v12+).

Выбрана по результатам исследования. Покрывает все требования:
- Кастомные узлы и рёбра (React-компоненты)
- Group nodes (sub-flows) для контейнеров команд
- Drag-and-drop узлов, создание рёбер через drag
- Minimap, zoom/pan, controls из коробки
- Нативная React-интеграция, TypeScript из коробки
- Совместимость с Tailwind для стилизации узлов

## 3. Структура Canvas

### 3.1 Общий вид

Один глобальный canvas. Все команды видны одновременно.

Каждая команда — визуальный контейнер (group node), внутри которого
расположены узлы-агенты. Контейнер показывает название команды и список workflow.

Связи (рёбра) существуют только **внутри workflow одной команды**.
Между командами связей нет.

### 3.2 Узлы (nodes)

Один узел = один агент.

Отображение узла:
- Имя агента
- Краткое описание (если есть)
- Цвет/иконка команды
- Индикатор "активен" (есть in_progress сессии)
- Маркер "Start" — если агент стартовый в каком-либо workflow

**Создание агентов:** кнопка **"+ Агент"** внутри group node команды.
Создаёт агента в БД и добавляет узел на canvas.

### 3.3 Рёбра (edges)

Ребро = переход от агента A к агенту B внутри workflow.

На ребре отображается:
- Название workflow (цветовая маркировка)
- Условие перехода (если есть)
- Имя промпта для агента-получателя
- Иконка: 🔒 (requires approval) или ⚡ (auto)

Рёбра направленные (стрелки). Циклы разрешены (A → B → A).

### 3.4 Создание команд

Кнопка **"+ Добавить команду"** на canvas:
1. Ввод названия и описания команды
2. Создаётся команда в БД
3. Появляется пустой group node на canvas
4. Внутри — кнопка "+ Агент" для добавления первого агента

## 4. Правая боковая панель

Открывается при клике на узел (агент) или ребро. Canvas остаётся интерактивным
(зум, скролл, выделение). Можно переключаться между узлами без закрытия панели.

### 4.1 Панель агента

Три вкладки: **Основное** / **Промпты** / **Handoff**.

#### Вкладка "Основное"

| Поле | Тип | Обязательное |
|------|-----|-------------|
| Name | text | да |
| Description | text | нет |
| System Prompt | textarea | нет |
| Allowed Tools | multi-select | нет |
| max_cycles | number (default: 3) | да |

`max_cycles` — максимальное количество раз, которое задача может вернуться
к этому агенту в рамках одного прогона workflow.

#### Вкладка "Промпты"

Список именованных промптов агента. Хранится в `agents.prompts` (JSONB).

```
[ Название промпта          ] [ Редактировать ] [ Удалить ]
  "Начать разработку"
  "Исправить замечания"
  "Провести code review"
[ + Добавить промпт ]
```

Редактор промпта:
- Название (string)
- Контент (textarea)

Переменные в промптах — подстановка динамических значений:
```
{{task_title}}          — название задачи (из Task)
{{task_description}}    — описание задачи (из Task)
```

Переменные берутся из контекста задачи. Агенты не передают данные друг другу
через переменные — вместо этого агент вызывает MCP tool для перехода,
передавая нужные параметры (см. раздел 6).

#### Вкладка "Handoff"

Настройка исходящих связей от этого агента. Сгруппированы по workflow.

```
Workflow: "Code Review Pipeline"
─────────────────────────────────────────────────────
Условие                  → Агент           Промпт              Auto
[ Code review passed ]   → [ QA       ▼]  [ Run tests    ▼]   [ ]    [ ✕ ]
[ Code review failed ]   → [ Developer▼]  [ Fix issues   ▼]   [ ]    [ ✕ ]
[ + Добавить правило ]

Workflow: "Quick Fix"
─────────────────────────────────────────────────────
(default)                → [ Reviewer ▼]  [ Quick review ▼]   [✓]    [ ✕ ]
[ + Добавить правило ]
```

Каждое правило:
- **Условие** — текстовое описание (агент интерпретирует через MCP tool)
- **Агент-получатель** — dropdown из агентов команды
- **Промпт** — dropdown из промптов агента-получателя
- **Auto** — checkbox: `requires_approval = false` (переход без подтверждения)

### 4.2 Панель ребра

При клике на ребро — быстрое редактирование:
- Условие перехода
- Промпт для получателя
- requires_approval toggle

## 5. Workflow (связка агентов)

### 5.1 Концепция

**Workflow** — именованная цепочка агентов внутри команды. Определяет порядок
обработки задачи: кто начинает, кому передаёт, при каких условиях.

Одна команда может иметь несколько workflow:
```
Team: "Development"
  ├── Workflow: "Full Review"     Developer → Reviewer → QA
  ├── Workflow: "Quick Fix"       Developer → Reviewer
  └── Workflow: "Hotfix"          Developer (один агент, без handoff)
```

При создании задачи пользователь выбирает: **Team → Workflow**.
Задача запускается по выбранному workflow.

### 5.2 Стартовый агент

Workflow имеет обязательное поле `starting_prompt` — промпт, который получает
первый агент при запуске задачи. Наличие заполненного `starting_prompt`
определяет стартового агента workflow.

Стартовый агент — агент, на который указывает `starting_agent_id` в Workflow.
Поле `starting_prompt` обязательно для заполнения.

На canvas стартовый агент отмечен маркером **"Start"**.

### 5.3 Завершение workflow

Агент без исходящих рёбер в данном workflow = конечный узел.
Когда этот агент завершает работу — задача переходит в статус `done`.

На canvas конечные узлы отмечены маркером **"End"** (вычисляется автоматически:
нет исходящих рёбер в данном workflow).

### 5.4 Создание workflow на canvas

1. Внутри group node команды — кнопка **"+ Workflow"**
2. Ввод названия workflow
3. Выбор стартового агента из агентов команды
4. Заполнение `starting_prompt` (обязательно)
5. Добавление рёбер (transitions) между агентами через drag или панель Handoff

### 5.5 Визуализация workflow на canvas

Каждый workflow имеет свой цвет. Рёбра одного workflow окрашены одним цветом.
Пользователь может фильтровать canvas по workflow (показать только один workflow).

## 6. Handoff через MCP

### 6.1 Замена текстовых handoff-блоков

Текущий механизм: агент генерирует текстовый блок `=== HANDOFF ===`,
backend парсит его. Проблемы: хрупкий парсинг, лишние токены на генерацию.

Новый механизм: Canvas-определённые переходы превращаются в **MCP tools**,
которые агент вызывает. MCP-сервер обрабатывает вызов и управляет переходом.

### 6.2 Генерация MCP tools из workflow

Для каждого агента Canvas генерирует набор MCP tools на основе его
исходящих рёбер в workflow:

```python
# Агент "Reviewer" в workflow "Full Review" имеет два исходящих ребра:
# 1. → QA (условие: "review passed")
# 2. → Developer (условие: "review failed")

# Генерируются MCP tools:
approve_and_forward_to_qa(comment: str)
return_to_developer(notes: str)
complete_task(summary: str)        # если агент — конечный узел
```

Агент видит эти tools в своём контексте и вызывает нужный по результату работы.

### 6.3 MCP-сервер Handoff

MCP-сервер `handoff-server` предоставляет:
- Tools: сгенерированные из workflow переходы (per-agent, per-workflow)
- При вызове tool: создание события перехода, проверка `requires_approval`
- Если `requires_approval = true`: WS-событие `approval_required` → ждёт approve
- Если `requires_approval = false`: автоматический переход к следующему агенту

### 6.4 Поток выполнения

```
1. Задача запускается → стартовый агент получает starting_prompt
2. Агент работает, вызывает MCP tool (например, forward_to_reviewer)
3. MCP-сервер проверяет requires_approval ребра
4. Если requires_approval:
   → WS-событие approval_required → алерт пользователю
   → Пользователь Approve/Reject
   → При Approve: создание sub-сессии для следующего агента
5. Если auto (requires_approval = false):
   → Автоматическое создание sub-сессии
6. Следующий агент получает промпт из шаблона ребра
7. Повторяется до конечного узла или max_cycles
```

### 6.5 Ограничение циклов

При каждом переходе MCP-сервер проверяет: сколько раз задача уже приходила
к целевому агенту. Если >= `agent.max_cycles`:
- Переход блокируется
- Задача переходит в статус `error`
- Пользователю отправляется алерт: "Max cycles reached for agent X"

## 7. Уведомления (алерты)

Система всплывающих уведомлений для событий, требующих действия пользователя.

### 7.1 Когда показывать алерт

| Событие | Алерт |
|---------|-------|
| Handoff ожидает approve | "Agent X → Agent Y: ожидает подтверждения" |
| max_cycles достигнут | "Task Z: цикл превышен для Agent X (error)" |
| Задача завершена (done) | "Task Z: выполнена" |
| Задача ошибка (error) | "Task Z: ошибка — описание" |

### 7.2 Где показывать

- Toast/snackbar в правом верхнем углу (на любой странице приложения)
- Если пользователь на другой странице — уведомление всё равно видно
- Уведомления доставляются через существующий WebSocket

## 8. Хранение данных

### 8.1 Новые сущности

Таблица `agent_links` удаляется. Вместо неё — две новые таблицы:

#### Таблица `workflows`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID PK | |
| name | varchar | Название workflow ("Full Review", "Quick Fix") |
| description | text, nullable | Описание |
| team_id | UUID FK → teams | Команда-владелец |
| starting_agent_id | UUID FK → agents | Стартовый агент |
| starting_prompt | text | Промпт для стартового агента (обязательный) |
| created_at | datetime | |
| updated_at | datetime | |

#### Таблица `workflow_edges`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID PK | |
| workflow_id | UUID FK → workflows | Какому workflow принадлежит |
| from_agent_id | UUID FK → agents | Откуда |
| to_agent_id | UUID FK → agents | Куда |
| condition | text, nullable | Условие перехода (текст, агент интерпретирует) |
| prompt_template | text, nullable | Шаблон промпта для to_agent |
| prompt_id | varchar, nullable | ID промпта из to_agent.prompts[] |
| order | int | Порядок проверки правил (при нескольких рёбрах) |
| requires_approval | bool, default true | Нужно ли подтверждение пользователя |
| created_at | datetime | |

#### Расширение таблицы `agents`

| Новое поле | Тип | Описание |
|------------|-----|----------|
| max_cycles | int, default 3 | Макс. возвратов к агенту в одном прогоне |
| position_x | float, nullable | X-координата на canvas |
| position_y | float, nullable | Y-координата на canvas |

### 8.2 Изменение Task

Таблица `tasks` (из TASKS_SPEC) меняется:

| Поле | Изменение |
|------|-----------|
| workflow_id | UUID FK → workflows (новое, заменяет starting_agent_id) |

При создании задачи: выбор Team → Workflow. `starting_agent_id` берётся
из workflow автоматически.

### 8.3 Авто-сохранение

Все изменения на canvas сохраняются автоматически при каждом действии:
- Перемещение узла → PATCH position_x/position_y
- Создание/удаление ребра → POST/DELETE workflow_edge
- Изменение настроек агента → PATCH agent
- Создание workflow → POST workflow

Debounce 500ms для позиций узлов (чтобы не спамить при drag).

## 9. Защита от изменений в runtime

### 9.1 Блокировка редактирования

Если по workflow выполняется задача (статус `in_progress` или `awaiting_user`):
- Workflow **заблокирован для редактирования** (рёбра, условия, промпты)
- На canvas отображается индикатор: "Workflow в работе"
- При попытке редактирования — уведомление:
  "Workflow используется задачей TASK-NNN. Завершите задачу для редактирования."

Разрешено при блокировке:
- Просмотр настроек (read-only)
- Перемещение узлов (позиция — визуальная, не влияет на runtime)

### 9.2 Блокировка удаления

Агент нельзя удалить, если:
- У него есть активные сессии (in_progress)
- Он входит в workflow с активной задачей

При попытке удаления — уведомление с причиной блокировки.

## 10. Валидация workflow

Canvas показывает визуальные предупреждения о проблемах в workflow:

| Проблема | Отображение | Блокирует запуск? |
|----------|-------------|-------------------|
| Workflow без стартового промпта | Ошибка (красная) | Да |
| Незавершённая цепочка (агент без рёбер, не конечный) | Предупреждение (жёлтое) | Нет |
| Недостижимый агент (нет входящих рёбер, не стартовый) | Предупреждение (жёлтое) | Нет |
| Команда без агентов | Информация (серое) | — |
| Команда без workflow | Информация (серое) | — |
| Агент без исходящих рёбер | Маркер "End" (нормально) | Нет |

Агент без исходящих рёбер — это конечный узел, задача → `done`. Это нормальное состояние.

## 11. Что не входит в Canvas-спеку

- Мульти-тенантность — будущее
- Версионирование workflow — будущее
- Экспорт/импорт workflow — будущее
- System Agent → team leaders — убрано из скоупа
- Undo/Redo на canvas — не в MVP
- Межкомандные связи — убрано из скоупа

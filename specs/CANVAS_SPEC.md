## Контекст

Canvas — визуальный редактор взаимосвязей агентов. Заменяет TeamPage.
На canvas видны все команды и их агенты, настраиваются handoff-цепочки,
условия переходов, промпты агентов и циклы.

Canvas определяет:
- Кто первый получает задачу в команде
- Кому и с каким промптом передаётся задача дальше
- При каких условиях происходит переход
- Сколько раз может зациклиться задача между агентами

## 3. Структура Canvas

### 3.1 Общий вид

Один глобальный canvas. На нём видны **все команды** и их агенты.

Команды визуально группируются — каждая команда в своём контейнере (group node).

Связи (рёбра) существуют только **внутри команды**. Между командами связей нет
(System Agent → team leaders убран из скоупа).

### 3.2 Узлы (nodes)

Один узел = один агент.

Отображение узла:
- Имя агента
- Краткое описание (если есть)
- Иконка команды / цвет команды
- Индикатор "активен" (есть in_progress сессии)

**[?] Как создаются агенты на canvas:**
- Вариант A: кнопка "+ Добавить агента" в группе команды → создаётся узел + агент в БД
- Вариант B: агенты создаются отдельно (форма), потом появляются на canvas автоматически

### 3.3 Рёбра (edges)

Ребро = handoff от агента A к агенту B.

На ребре отображается:
- Условие перехода (если есть)
- Промпт который получит агент B (имя промпта)

Рёбра направленные (стрелки). Возможны циклы (A → B → A).

## 4. Модальное окно агента

Открывается по клику на узел. Содержит все настройки агента.

**[?] Структура модала:** одно окно или вкладки? Предполагаемые вкладки:
- **Основное** — name, description, system_prompt, allowed_tools
- **Промпты** — список именованных промптов (редактор)
- **Handoff** — настройка связей (кому передаёт, условия, промпты)

### 4.1 Вкладка "Основное"

Поля:
- Name (обязательный)
- Description (опциональный)
- System Prompt (textarea)
- Allowed Tools (multi-select или список строк)

### 4.2 Вкладка "Промпты"

Список именованных промптов агента. Хранится в `agents.prompts` (JSONB).

Каждый промпт:
```
[ Название промпта          ] [ Редактировать ] [ Удалить ]
  "Начать разработку"
  "Исправить замечания"
  "Провести code review"
```

Редактор промпта:
- Название (string)
- Контент (textarea с поддержкой переменных)

**Переменные в промптах** — подстановка динамических значений:
```
{{task_title}}          — название задачи
{{task_description}}    — описание задачи
{{reviewer_notes}}      — произвольные данные от предыдущего агента
```

**[?] Источник переменных:** как агент передаёт данные следующему через переменные?
Варианты:
- A: агент пишет специальный блок в конце ответа (как handoff-блок сейчас)
- B: пользователь заполняет переменные вручную при approve
- C: переменные парсятся из последнего сообщения агента по шаблону

### 4.3 Вкладка "Handoff"

Настройка исходящих связей от этого агента.

Список правил:

```
Условие                     Агент-получатель     Промпт
─────────────────────────────────────────────────────────────
[ Тесты прошли: да    ]  →  [ Reviewer      ▼]  [ Code Review ▼]   [ Удалить ]
[ Тесты прошли: нет   ]  →  [ Developer     ▼]  [ Исправить   ▼]   [ Удалить ]
[ + Добавить правило ]
```

**[?] Формат условий:** как задаётся условие перехода?
- A: свободный текст (агент сам интерпретирует в handoff-блоке)
- B: структурированные условия (dropdown: статус/результат = значение)
- C: первый этап — свободный текст, потом структурированные

**Циклы:** правило может ссылаться на агента из предыдущего уровня (A → B → A) — это ок.

## 5. Ограничение циклов

Каждый агент имеет настройку `max_cycles` — максимальное количество раз,
которое задача может вернуться к этому агенту в рамках одного прогона.

- Default: 3
- Настраивается в модале агента (вкладка "Основное" или отдельное поле)
- Хранится в `agents.config` JSON или отдельном поле

При достижении лимита: задача не возвращается к агенту, граф завершается
(задача переходит в `error` или `done` — **[?] что именно?**).

## 7. Влияние на runtime

### 7.1 Запуск задачи

При старте задачи canvas определяет `starting_agent_id`.
Первый агент = агент без входящих рёбер в команде (или явно помеченный как стартовый).

**[?] Если несколько агентов без входящих рёбер:** ошибка валидации или пользователь
выбирает вручную при создании задачи (уже предусмотрено в TaskCreate)?

### 7.2 Handoff в graph_service.py

`gate_node` при approve создаёт sub-сессию для следующего агента.
Следующий агент сейчас берётся из `agent_links` через `get_agent_handoff_targets`.

С Canvas: логика выбора следующего агента усложняется — нужно учитывать условия.

**[?] Кто вычисляет условие перехода:**
- A: граф смотрит на последний ответ агента и матчит условие
- B: пользователь выбирает при approve (approve к конкретному агенту)
- C: агент сам указывает target в handoff-блоке (как сейчас), canvas только валидирует
